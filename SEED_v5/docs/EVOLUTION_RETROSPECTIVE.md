# SEED v4 進化プロジェクト: 振り返りと未来へのガイド

このドキュメントは、Ops OS (v1.0) から SEED v4 への進化プロセスにおいて、**「なぜ時間がかかったのか（反省）」** と **「今後どう運用すべきか（未来）」** を記した羅針盤です。

## 1. 振り返り: なぜ v3.1.0 → v4 化は難航したのか？

今回の移行作業が長時間化した主な要因は、以下の4点に集約されます。

### 🚧 A. 「固有アプリ」と「汎用キット」の混同 (Context Mixing)
*   **事象**: Ops OS v1.0 の成果物 (`CHANGELOG.md`, `DEPLOYMENT.md`) をそのまま v4 に持ち込んでしまった。
*   **影響**: 「汎用ツールなのに Ops OS の履歴が入っている」状態になり、手戻り（クリーンアップ作業）が発生した。
*   **教訓**: **「中身（コンテンツ）」と「枠組み（フレーム）」を最初から明確に分けるべき**でした。

### 📉 B. 「進化」による「哲学」の消失 (Philosophy Loss)
*   **事象**: v4 の新機能 (Rubric, Security) を強調するあまり、`SEED_GUIDE.md` を全面的に書き換え、v3 が持っていた重要な哲学（Hexagon, Knowledge Loop）を消してしまった。
*   **影響**: 「前の良さが消えている」という指摘を受け、マージ（統合）作業が発生した。
*   **教訓**: 進化とは上書きではなく **「継承と拡張」** であるべきです。

### 💾 C. ファイル操作の手詰まり (Technical Friction)
*   **事象**: Windows 環境での `move` や `copy` コマンドが対話モード（Waiting）になり、何度もプロセスが停止した。
*   **影響**: 単純なファイル移動に数ターンを消費した。
*   **教訓**: 大量ファイル操作はコマンドに頼らず、**「新規作成 (Write) & 古い削除」** の安全策を採るべきです。

### 🌐 D. 言語の不一致 (Localization Gap)
*   **事象**: 思考は日本語で行っていたが、生成されたドキュメントが英語だった。
*   **影響**: 「日本語にして」という修正ターンが発生した。
*   **教訓**: SEED は **"Japanese First"** の憲章を持つため、最初から日本語で出力すべきでした。

## 2. 実践ガイド: SEED v4 を正しく稼働・進化させるために

今後、SEED v4 を使って新しいアプリ（日記帳など）を作り、さらに SEED v5 へと進化させていくための 3つの鉄則です。

### 🚀 ルール1: 「インスタンス化」から始める (Instantiate, Don't Modify)
SEED v4 フォルダそのものを開発場所にしてはいけません。
*   **× Bad**: `SEED_v4/` の中でコードを書き始める。
*   **◎ Good**: `SEED_v4/` を `MyNewApp/` にコピーしてから始める。
*   **理由**: SEED v4 自体が汚染されると、次のプロジェクトで「ゴミ掃除」から始めなければならなくなります（今回の反省）。

### 🧠 ルール2: 最初の一手は "Rubric Tuning" (Brain First)
コードを書く前に、必ず **Rubric Engine の基準** を決めてください。
*   汎用版のまま使うと、日記アプリなのに「敬語を使え」と怒られます（今回の日記デモなど）。
*   `ops/rubric_engine.js` のプロンプトを、そのアプリの「キャラ設定」に合わせて書き換えること。これが v4 開発の **「心臓部」** です。

### 🌾 ルール3: 収穫は「汎用的なもの」だけ (Selective Harvest)
プロジェクト終了後、SEED に機能を還流（バックポート）する際は、厳しく選別してください。
*   **× 還流しない**: 日記アプリ専用の UI、特定DBのデータ。
*   **◎ 還流する**: 便利だった汎用関数 (`utils.js`)、バグ修正、セキュリティ設定。
*   **手順**:
    1. プロジェクト完了。
    2. `EVOLUTION_PROTOCOL.md` (成長プロンプト) を実行。
    3. 「全プロジェクトで使えるか？」を自問し、Yes のものだけを `SEED_vNext` に取り込む。
